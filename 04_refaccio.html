Refacció (refactoring)

Modificar l'estructura interna del codi sense canviar-ne el comportament extern. L'objectiu és fer el codi més llegible, més curt, més mantenible.

✓ Què SÍ és refacció
  • Extreure un fragment repetit a un mètode nou
  • Donar noms millors a variables i mètodes
  • Substituir un número màgic per una constant amb nom
  • Eliminar codi duplicat
  • Reorganitzar mètodes entre classes
✗ Què NO és refacció
  • Afegir noves funcionalitats
  • Corregir bugs (això és correcció, no refacció)
  • Optimitzar rendiment canviant l'algorisme
  • Reescriure des de zero
⚠️ Regla d'or

Si abans i després de refactoritzar passes els mateixos tests amb el mateix resultat, ho has fet bé. Si algun test canvia, és que has alterat el comportament — això ja no és refacció.

Tècniques de refacció (les que cauen a la PAF)

1️⃣ Extreure mètode

Quan veus un bloc de codi que fa una cosa concreta, mou-lo a un mètode separat.

Abans:

if (tipus.equals("MONO")) {
    base = hores * 250;
    if (hores > 7) base += 50;
    if (temp > 25) base -= base * 0.05;
    return base;
}

Després:

if (tipus.equals("MONO")) {
    return calcularMono(hores, temp);
}

private double calcularMono(double hores, int temp) {
    double base = hores * 250;
    if (hores > 7) base += 50;
    if (temp > 25) base -= base * 0.05;
    return base;
}
2️⃣ Extreure constant

Els "números màgics" (250, 0.05, 25...) no diuen res. Substitueix-los per constants amb nom.

private static final double POTENCIA_MONO = 250;
private static final double PERDUA_CALOR_MONO = 0.05;
private static final int TEMP_LIMIT = 25;
private static final int HORES_BONUS = 7;
private static final double BONUS_MONO = 50;

Avantatge: si canvies TEMP_LIMIT a 30, només toques un lloc.

3️⃣ Reanomenar (rename)

Donar noms descriptius. x, aux, temp1 → preuFinalAmbDescompte, nombreDeIntentos, etc.

L'IDE ho fa automàticament: botó dret → Refactor → Rename. Canvia totes les referències d'una sola vegada.

4️⃣ Moure mètode/atribut

Si un mètode usa més els atributs d'una altra classe que els de la seva pròpia, és senyal que viu a la classe equivocada.

5️⃣ Extreure variable

Per a expressions complicades, extreu el resultat a una variable amb nom descriptiu.

// Abans
if ((preuBase * 1.21) - descompte > 1000) ...

// Després
double totalAmbIVA = preuBase * IVA_FACTOR;
double totalFinal = totalAmbIVA - descompte;
if (totalFinal > LIMIT_VIP) ...
6️⃣ Eliminar duplicació amb herència/polimorfisme

Si tens tres branques que fan gairebé el mateix amb petits canvis, considera crear una classe pare amb el patró comú i classes filles per a les diferències.

Exemple complet — basat en l'EAC3

Imagina un sistema que calcula la producció d'un panell solar segons tipus (MONO, POLY, FILM), hores de sol i temperatura.

Codi original (sense refactoritzar)

public class GestorEnergia {

    public static double calcularProduccio(String tipus, double hores, int temp) {
        double base = 0;
        double rendiment = 0;

        if (tipus.equals("MONO")) {
            base = hores * 250;
            if (hores > 7) base += 50;
            if (temp > 25) rendiment = base - (base * 0.05);
            else rendiment = base;
            System.out.println("Mono: " + rendiment);
            return rendiment;
        } else if (tipus.equals("POLY")) {
            base = hores * 200;
            if (hores > 7) base += 30;
            if (temp > 25) rendiment = base - (base * 0.08);
            else rendiment = base;
            System.out.println("Poly: " + rendiment);
            return rendiment;
        } else if (tipus.equals("FILM")) {
            base = hores * 150;
            if (temp > 25) rendiment = base - (base * 0.02);
            else rendiment = base;
            System.out.println("Film: " + rendiment);
            return rendiment;
        } else {
            System.out.println("Error");
            return -1;
        }
    }
}

Olors detectades

  1. Codi duplicat: el patró "calcular base + bonus + pèrdua per calor + log" es repeteix 3 vegades.
  2. Números màgics: 250, 200, 150, 50, 30, 7, 25, 0.05, 0.08, 0.02. Cap diu què és.
  3. Mètode llarg: 25+ línies, branques complexes.
  4. Strings literals com a identificadors: "MONO", "POLY", "FILM" repartits per tot.
  5. Retorn de -1 com a error: antipatron, millor llançar excepció.

Codi refactoritzat

public class GestorEnergia {

    // === REFACCIÓ 1: extreure constants (eliminar números màgics) ===
    private static final double POT_MONO = 250;
    private static final double POT_POLY = 200;
    private static final double POT_FILM = 150;
    private static final double BONUS_MONO = 50;
    private static final double BONUS_POLY = 30;
    private static final double PERDUA_MONO = 0.05;
    private static final double PERDUA_POLY = 0.08;
    private static final double PERDUA_FILM = 0.02;
    private static final int HORES_LIMIT_BONUS = 7;
    private static final int TEMP_LIMIT_PERDUA = 25;

    public static double calcularProduccio(String tipus, double hores, int temp) {
        switch (tipus) {
            case "MONO":
                return calcular(hores, temp, POT_MONO, BONUS_MONO, PERDUA_MONO, "Monocristal·lí");
            case "POLY":
                return calcular(hores, temp, POT_POLY, BONUS_POLY, PERDUA_POLY, "Policristal·lí");
            case "FILM":
                return calcular(hores, temp, POT_FILM, 0, PERDUA_FILM, "Capa Fina");
            default:
                System.out.println("Error: tipus desconegut");
                return -1;
        }
    }

    // === REFACCIÓ 2: extreure mètode (eliminar duplicació) ===
    private static double calcular(double hores, int temp,
                                     double potencia, double bonus,
                                     double perdua, String nom) {
        double base = hores * potencia;
        if (hores > HORES_LIMIT_BONUS) base += bonus;
        double rendiment = (temp > TEMP_LIMIT_PERDUA) ? base - (base * perdua) : base;
        System.out.println(nom + ": " + rendiment + " watts");
        return rendiment;
    }
}
Tècniques aplicades (mínim 2 que es demanen a l'EAC3)
  1. Extreure constants: tots els números màgics tenen nom ara.
  2. Extreure mètode: el patró comú de càlcul ara és calcular(), cridat 3 vegades.
  3. Bonus: substitució de if-else-if per switch (més llegible).
  4. Bonus: operador ternari ?: on hi havia un if/else trivial.

El comportament és idèntic al codi original: per cada tipus, els valors numèrics finals són els mateixos.

⚠️ Diferència crítica

Comentaris normals: // ... o /* ... */ — només per a humans que llegeixin el codi.

Comentaris Javadoc: /** ... */ (amb doble asterisc) — l'eina javadoc els processa per generar HTML.

Errors típics a la refacció

❌ Canviar el comportament accidentalment

Refactoritzes i el codi ja no fa el mateix. Sempre executa els tests abans i després per assegurar-te que el resultat és idèntic.

❌ Sobre-refactoritzar (over-engineering)

Crear 5 nivells d'herència per un sistema senzill. La refacció ha de simplificar, no complicar. Si el codi original és curt i clar, potser no cal refactoritzar.

❌ Refactoritzar i afegir features alhora

Si fas les dues coses al mateix commit, quan algun test trenqui no sabràs si és per la refacció o per la nova funcionalitat. Separa els canvis.

⚠️ A la PAF (escrivint a paper)

No t'has de saber les ordres exactes d'Eclipse (Right-click → Refactor → ...). N'hi ha prou amb:

  • Identificar les olors al codi original
  • Saber el nom de la tècnica que aplicaries
  • Escriure el codi resultant (encara que sigui simplificat)
  • Comentar al costat quina tècnica has aplicat: // extracció de mètode
💡 Truc PAF

L'enunciat sol demanar "com a mínim dues tècniques". Les més segures per aplicar i fàcils d'explicar són:

  1. Extreure constants (números màgics)
  2. Extreure mètode (codi duplicat)

Aquestes dues quasi sempre apliquen a qualsevol codi raonablement complex.

Quiz — Refacció (10 preguntes)

1. Refacció (refactoring) significa:
Refacció = millorar l'estructura sense canviar el que fa. Si afegeixes funcionalitats o corregeixes bugs, ja no és refacció pura.
2. Quin patró s'usa per convertir un fragment de codi en un mètode independent?
Extract Method: agafes un bloc de codi i el converteixes en un mètode amb nom propi.
3. Un "número màgic" es resol amb:
Substitueix el literal pel nom d'una constant: private static final double IVA = 0.21;
4. Quin d'aquests NO és un code smell?
Una classe amb pocs atributs és perfectament normal. Les altres opcions són olors clàssiques que indiquen necessitat de refacció.
5. Per saber si la refacció no ha trencat res, l'eina més útil és:
Si abans i després els tests passen amb els mateixos resultats, la refacció és correcta. Compilar només garanteix que el codi és sintàcticament vàlid.
6. Un comentari Javadoc comença amb:
El doble asterisc inicial (/**) és el que els distingeix de comentaris normals i fa que l'eina javadoc els processi.
7. L'etiqueta Javadoc per documentar un paràmetre és:
Format estàndard: @param nomVariable Descripció.
8. Què NO és correcte fer durant una sessió de refacció?
Refacció i afegir features han d'anar en commits separats. Si barreges, no podràs identificar quin canvi ha trencat un test.
9. Tenim aquest codi: if (preu > 1000) descompte = preu * 0.1; El número màgic és:
Tots dos haurien de ser constants: LIMIT_DESCOMPTE_VIP i PERCENTATGE_DESCOMPTE.
10. Si demanes a un assistent d'IA suggeriments de refacció i et proposa 5 canvis, com hauries d'actuar?
L'EAC3 demana exactament això: "anàlisi crítica" — quins acceptes i quins descartes, i per què. La IA pot suggerir patrons innecessaris per a codi simple.
