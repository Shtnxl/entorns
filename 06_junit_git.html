06 · JUnit + Git — interpretació

← Índex

Aquest tema cau només com a test

A la PAF no hauràs d'escriure tests JUnit ni executar ordres Git. Hauràs d'interpretar codi i ordres ja escrits, i identificar el que fan.

El que es pot preguntar
  • "Què fa l'anotació @Test?"
  • "Aquest assertEquals(5, x) falla? Per què?"
  • "Quina ordre Git crea una branca nova?"
  • "Què passa si fas git merge i hi ha un conflicte?"
  • "Per a què serveix el fitxer .gitignore?"

JUnit 5 — Anotacions clau

AnotacióQuè faQuan s'executa
@TestMarca un mètode com a provaCada vegada que executes els tests
@BeforeEachCodi que s'executa abans de cada testAbans de cada @Test
@AfterEachCodi que s'executa després de cada testDesprés de cada @Test
@BeforeAllUna sola vegada al principi (mètode static)Abans del primer test
@AfterAllUna sola vegada al final (mètode static)Després de l'últim test
@DisabledSalta aquest test (no s'executa)Mai
@DisplayName("...")Nom amigable que apareix als resultats—

Exemple complet

package gestio;

import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.BeforeEach;
import static org.junit.jupiter.api.Assertions.*;

public class TestControlVol {

    private ControlVol cv;

    @BeforeEach
    void setUp() {
        cv = new ControlVol();    // abans de cada test
    }

    @Test
    void testBateriaOptima() throws Exception {
        String resultat = ControlVol.alertaBateria(95);
        assertEquals("Bateria òptima", resultat);
    }

    @Test
    void testValorInvalid() {
        assertThrows(BateriaIncorrectaException.class, () -> {
            ControlVol.alertaBateria(-5);
        });
    }
}
💡 Convenció de noms

El nom de la classe de test sol començar per Test (TestControlVol) o acabar amb Tests. Cada mètode @Test hauria d'expressar què prova: testValorInvalid, testBateriaOptima...

Mètodes assert de JUnit 5

Són mètodes estàtics de la classe org.junit.jupiter.api.Assertions.

MètodeQuè comprovaExemple
assertEquals(esperat, actual) Que els dos valors siguin iguals assertEquals(5, x)
assertNotEquals(no_esperat, actual) Que NO siguin iguals assertNotEquals(0, divisor)
assertTrue(condició) Que sigui true assertTrue(llista.isEmpty())
assertFalse(condició) Que sigui false assertFalse(usuari.isAdmin())
assertNull(obj) Que l'objecte sigui null assertNull(resultat)
assertNotNull(obj) Que l'objecte NO sigui null assertNotNull(usuari)
assertThrows(Tipus.class, () -> ...) Que el codi llanci una excepció Vegeu exemple a baix
assertArrayEquals(a, b) Que dos arrays siguin iguals element a element assertArrayEquals(new int[]{1,2}, x)
fail("missatge") Forçar que la prova falli sempre fail("No s'hauria d'arribar aquí")

L'ordre dels arguments d'assertEquals

⚠️ Sempre: esperat primer, actual després
assertEquals(10, calcular(5));
//           ↑esperat    ↑resultat real

Si t'equivoques d'ordre, el test funciona igual però el missatge d'error queda invertit i confús ("expected 5, but was 10" quan en realitat era al revés).

assertThrows — el important per a la PAF

@Test
void testValorNegatiu() {
    assertThrows(BateriaIncorrectaException.class, () -> {
        ControlVol.alertaBateria(-5);
    });
}
Llegir aquest assert

Diu: "esperem que el codi dins la lambda llanci una BateriaIncorrectaException".

  • Si llança aquesta excepció → test OK
  • Si llança una altra excepció → test FALLA
  • Si no llança cap excepció → test FALLA
❌ Error típic

Si proves un mètode que llança una excepció checked (Exception, no RuntimeException), has de marcar el test amb throws:

@Test
void testBateriaNormal() throws Exception {  // ← necessari!
    assertEquals("Operació normal", ControlVol.alertaBateria(70));
}

Git — ordres bàsiques

OrdreQuè fa
git initInicialitza un repositori nou a la carpeta actual
git statusMostra fitxers modificats, staged, no rastrejats
git add <fitxer>Afegeix un fitxer a l'àrea de staging

git commit -m "missatge"Crea un commit amb els canvis staged
git logMostra la història de commits
git diffMostra els canvis pendents

El cicle típic

Carpeta de treball  →  Staging area  →  Repositori local
    (modifies)         (git add)         (git commit)

Estats dels fitxers

EstatSignificat
UntrackedGit no en sap res, mai s'ha fet add
ModifiedModificat però encara no afegit a staging
StagedAfegit amb git add, llest per commit
CommittedGuardat al repositori

El fitxer .gitignore

# Comentaris amb #
*.class
*.log
target/
.DS_Store
node_modules/

Indica a Git quins fitxers o patrons ha d'ignorar (no rastrejar). Típicament: binaris generats, logs, configuracions locals, secrets.

⚠️ Una vegada un fitxer està rastrejat...

Afegir-lo al .gitignore NO el treu de Git. Cal fer git rm --cached <fitxer> per fer que Git deixi de seguir-lo.

Tipus de merge

Fast-forward

Si la branca destí no té commits propis des que es va separar la branca origen, Git només "mou el punter" cap endavant. No crea cap commit nou de merge.

master   A───B
              \\
branca         C───D

Després de "git merge branca" des de master:

master   A───B───C───D  (només mou el HEAD endavant)
Merge amb 3 vies (3-way merge)

Quan les dues branques tenen commits propis, Git crea un commit de merge nou que combina les dues històries.

master   A───B───E
              \\       \\
branca         C───D───M (commit de merge)

Conflictes de merge

Un conflicte es produeix quan dues branques han modificat les mateixes línies d'un mateix fitxer.

Cas pràctic (com a l'EAC3)

master: tens private int autonomia;

branca_autonomia: canvies a private double autonomia;

branca_nom: canvies el mateix atribut a private int tempsVolMaxim;

Quan intentes fusionar les dues branques a master, Git no sap quin canvi guardar i marca el conflicte.

Com es veu un conflicte al fitxer

// Dron.java
<<<<<<< HEAD
    private double autonomia;
=======
    private int tempsVolMaxim;
>>>>>>> branca_nom
  • HEAD = la versió de la branca actual (master)
  • branca_nom = la versió que vols incorporar
  • Cal editar manualment el fitxer, decidir quin codi va i esborrar els marcadors <<<, ===, >>>

Procediment per resoldre

  1. git status per veure quins fitxers tenen conflicte.
  2. Obre el fitxer i edita'l manualment.
  3. Deixa només el codi que vols mantenir (potser una barreja dels dos).
  4. Esborra els marcadors <<<, ===, >>>.
  5. git add <fitxer> per marcar com a resolt.
  6. git commit per completar el merge.
Cas EAC3 — la solució "barreja"

Volies mantenir tots dos canvis: el tipus double de branca_autonomia I el nom tempsVolMaxim de branca_nom. La resolució manual seria:

    private double tempsVolMaxim;

Mostra que Git no decideix per tu — només marca el conflicte. La decisió és humana.

Tipus de canvis que NO causen conflicte

  • Dues branques modifiquen línies diferents del mateix fitxer → Git fusiona automàticament.
  • Dues branques modifiquen fitxers diferents → cap problema.
  • Una branca afegeix un fitxer nou que no existeix a l'altra → s'afegeix tal com ve.

Quiz — JUnit + Git (15 preguntes)

1. L'anotació @Test de JUnit serveix per:
Cada mètode amb @Test serà executat per JUnit quan corris la suite.
2. Què comprova assertEquals(5, x)?
Primer paràmetre = esperat, segon = real. Comprova igualtat estricta.
3. @BeforeEach s'executa:
Permet inicialitzar variables abans de cada test, garantint que cada prova és independent. @BeforeAll és el que s'executa una vegada.
4. assertThrows(IllegalArgumentException.class, () -> metode(-5)) passa si:
El test verifica que es llanci EXACTAMENT aquell tipus d'excepció. Si llança una altra, o no llança res, el test falla.
5. L'ordre per inicialitzar un repositori Git nou és:
git init crea el directori .git/ a la carpeta actual.
6. Per saber quins fitxers tenen canvis pendents, l'ordre és:
git status mostra fitxers modificats, staged i untracked. git log mostra commits passats, no canvis actuals.
7. Què és el HEAD?
HEAD apunta sempre a on ets ara. Després d'un nou commit, HEAD avança a aquest nou commit.
8. Què és un conflicte de merge?
Cal resoldre'l manualment editant el fitxer i decidint quins canvis es queden.
9. El fitxer .gitignore serveix per:
S'hi posen patrons com *.class, target/, .DS_Store per evitar pujar binaris o configs locals.
10. Per crear i situar-se a una branca nova alhora:
git branch sense -b només crea, no mou. checkout -b fa les dues coses.
11. Quan Git fa un merge "fast-forward":
Passa quan la branca destí no té commits propis des que es va separar. És el merge més senzill possible.
12. Què fa git add .?
El punt vol dir "tots els fitxers de la carpeta actual i subcarpetes". git add mai crea commits, només prepara fitxers per al commit.
13. En un fitxer amb conflicte, els marcadors <<<<<<< HEAD indiquen:
Després de <<< HEAD i fins a === hi ha la teva versió. De === a >>> hi ha la versió de l'altra branca.
14. Què fa assertNotNull(obj)?
Si obj és null, el test falla. Útil per verificar que un constructor o factory mètode realment crea un objecte.
15. La branca principal d'un projecte Git tradicional s'anomena:
Tradicionalment "master". Els repositoris moderns sovint l'anomenen "main" com a alternativa.
