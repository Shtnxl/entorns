Proves: per què i com

Provar és demostrar que el codi fa el que ha de fer i que no fa el que NO ha de fer. No pots provar-ho tot, però amb tècniques sistemàtiques pots cobrir el que importa amb poques proves.

Tipus de proves segons abast

SistemaTota l'aplicacióSelenium, Postman

A la PAF

El nivell que cau és proves unitàries: classes d'equivalència, valors límit, complexitat ciclomàtica i camins independents.

Caixa negra vs caixa blanca

Caixa negraCaixa blanca
Coneixes el codi intern?NO, només l'especificacióSÍ, mires el codi
Què miresEntrades → sortidesEstructura interna, branques

💡 Truc per recordar

Negra = no veus l'interior, només l'exterior (entrades/sortides). Blanca = és transparent, veus tot el codi.

⚠️ A la PAF

Si la pregunta diu "donada aquesta especificació" → caixa negra (equivalència, límits).

Si la pregunta diu "donat aquest codi" → caixa blanca (complexitat, camins).

Classes d'equivalència

Idea: si dos valors fan que el codi es comporti igual, no cal provar-los tots dos. Agrupes les entrades en classes i proves una mostra de cada.

Exemple: descompte segons edat

Especificació:

  • 0-17 anys: descompte del 50%
  • 18-64 anys: sense descompte
  • 65+ anys: descompte del 30%
  • Edat negativa o > 120: error

Classes que en surten

#ClasseRangTipusValor de prova
1Menor d'edat[0, 17]vàlida10
2Adult[18, 64]vàlida35
3Sènior[65, 120]vàlida72
4Negatiu(-∞, -1]invàlida-5
5Massa gran[121, +∞)invàlida200
Resultat

Amb només 5 proves hem cobert totes les zones de comportament diferent. Si en lloc d'això haguéssim provat edats 1, 2, 3, 4... seria absurd: les edats 1-17 fan el mateix.

⚠️ Regla per identificar classes

Una classe per cada interval on el codi es comporta de la mateixa manera. Si el comportament canvia (descompte vs error, una branca vs una altra), és una nova classe.

Classes vàlides i invàlides

L'enunciat de la PAF moltes vegades demana com a mínim una de cada tipus: vàlides (entrades que el codi accepta) i invàlides (entrades que ha de rebutjar amb error).

Valors límit (boundary values)

Els bugs es concentren als extrems de les classes. Si una classe és [18, 64], els valors crítics són 18, 64, i els seus veïns immediats (17, 19, 63, 65).

Per a cada límit, prova 3 valors

  1. Just per sota del límit
  2. El límit exacte
  3. Just per sobre del límit

Exemple amb el mateix problema d'edat

LímitPer sotaLímitPer sobre
Inici menor-1 (invàlid)0 (vàlid)1 (vàlid)
Final menor16 (vàlid)17 (vàlid)18 (canvi)
Final adult63 (vàlid)64 (vàlid)65 (canvi)
Final sènior119 (vàlid)120 (vàlid)121 (invàlid)
Per què funciona això

Els errors típics són confondre < amb <=, o > amb >=. Provar els valors als límits els detecta immediatament.

Exemple: Si el codi diu if (edat < 18) en lloc de if (edat <= 17), només la prova amb edat=17 detectarà la diferència entre les dues versions del codi.

⚠️ Combinació equivalència + límit

Quasi sempre es fan totes dues juntes: identifiques les classes (equivalència) i després afegeixes proves als límits de cada classe. És la combinació estàndard de proves de caixa negra.

Complexitat ciclomàtica V(G)

Una mètrica que diu quantes proves diferents calen com a mínim per cobrir tots els camins lògics del codi.

Fórmula

V(G) = aristes − nodes + 2

On el graf G és el diagrama de flux del codi: cada instrucció és un node, cada salt una aresta.

Atall pràctic: comptar punts de decisió + 1

Forma equivalent i molt més ràpida:

V(G) = (nombre de punts de decisió) + 1

Un punt de decisió és cadascun d'aquests:

  • Cada if
  • Cada else if
  • Cada case d'un switch
  • Cada while / for / do-while
  • Cada operador lògic && o || dins d'una condició
⚠️ L'operador lògic compta!
if (a > 0 && b < 100) { ... }

Aquesta línia té 2 punts de decisió: el if i el &&. Internament Java fa curtcircuit, i això crea dos camins (si a>0 és fals, ja no avalua b<100).

Exemple pas a pas

public String classificar(int nota) {       // node 1 (entrada)
    if (nota < 0 || nota > 10) {           // 2 decisions (if + ||)
        return "Error";
    }
    if (nota < 5) {                          // 1 decisió
        return "Suspès";
    } else if (nota < 7) {                   // 1 decisió
        return "Aprovat";
    } else {
        return "Notable";
    }
}

Càlcul

Punts de decisió: 1 (if) + 1 (||) + 1 (if) + 1 (else if) = 4

V(G) = 4 + 1 = 5

Interpretació

Calen com a mínim 5 proves per cobrir tots els camins lògics. Una pràctica habitual diu que codi amb V(G) > 10 és massa complex i s'hauria de refactoritzar.

Camins independents

Un cop saps V(G), has de trobar N camins diferents pel diagrama de flux, on N = V(G). Cadascun ha d'incloure almenys una aresta nova respecte els anteriors.

Procediment

  1. Dibuixa el diagrama de flux (nodes + aristes).
  2. Numera els nodes (1, 2, 3...).
  3. Comença pel camí "directe" (entrada → primera sortida).
  4. Per cada nou camí, fes un canvi a una decisió que abans havies fet de l'altra manera.
  5. Continua fins tenir V(G) camins, cadascun amb almenys una nova aresta.

Exemple amb el classificar() anterior

V(G) = 5 → calen 5 camins:

#CamíCas de provaSortida
1nota fora rang (negativa)nota = -5"Error"
2nota fora rang (massa alta)nota = 15"Error"
3nota vàlida, suspesanota = 3"Suspès"
4nota vàlida, aprovadanota = 6"Aprovat"
5nota vàlida, notablenota = 8"Notable"
💡 Truc PAF

Per cada camí, explicita: (a) per quins nodes passa, (b) quin valor d'entrada el provoca, (c) quina sortida produeix.

⚠️ Diferència entre camins i casos de prova

Un camí independent és un recorregut pel graf. Un cas de prova és l'entrada concreta que el produeix. Per cada camí cal un cas de prova, però un mateix camí pot tenir múltiples casos.

Ex: nota=3 i nota=4 segueixen el mateix camí ("Suspès"). Només cal un d'ells per al test.

Exemple complet (estil EAC3 / PAF)

Especificació

Un mètode calcularPreu(int unitats, boolean esVip) calcula el preu d'una comanda:

  • Unitats < 1: llança IllegalArgumentException
  • Unitats 1-9: preu = unitats × 10
  • Unitats 10-49: preu = unitats × 9 (descompte volum)
  • Unitats ≥ 50: preu = unitats × 8 (descompte gran)
  • Si esVip = true: aplicar 10% addicional de descompte al final

3. Caixa blanca: complexitat ciclomàtica

public double calcularPreu(int u, boolean vip) {
    if (u < 1) {                          // decisió 1
        throw new IllegalArgumentException();
    }
    double preu;
    if (u < 10) preu = u * 10;             // decisió 2
    else if (u < 50) preu = u * 9;         // decisió 3
    else preu = u * 8;
    if (vip) preu *= 0.9;                  // decisió 4
    return preu;
}

Punts de decisió: 4 → V(G) = 5

4. Camins independents

  1. u < 1 → excepció
  2. u < 10, no vip → preu × 10
  3. u en [10, 49], no vip → preu × 9
  4. u ≥ 50, no vip → preu × 8
  5. qualsevol u vàlid, vip → s'aplica 0.9

5. Valors límit

  • 0, 1, 2 → al voltant del límit inferior vàlid
  • 9, 10, 11 → al voltant del canvi de tarifa volum
  • 49, 50, 51 → al voltant del canvi de tarifa gran

Combinant les 3 tècniques (equivalència + límits + camins) cobreixes el codi de manera robusta.

Quiz — Proves (12 preguntes)

1. Les proves de caixa negra es basen en:
"Negra" = no veus dins. Provem entrades → comprovem sortides, sense saber com es calculen internament.
2. Les classes d'equivalència serveixen per:
L'objectiu és reduir el nombre de proves: si edats 1-17 fan el mateix, no cal provar les 17 — n'hi ha prou amb una.
3. Per a una classe vàlida de l'interval [10, 99], els valors límit típics són:
Per a cada límit: just per sota, límit, just per sobre. Així detectes errors típics de < vs <=.
4. La fórmula de complexitat ciclomàtica V(G) és:
V(G) = E − N + 2 on E són aristes i N nodes. Equivalent: nombre de punts de decisió + 1.
5. Aquest codi té quina complexitat ciclomàtica?
if (a > 0) {
    if (b > 0) x();
    else y();
}
2 punts de decisió (2 ifs) + 1 = 3. Camins: (a≤0), (a>0 i b>0), (a>0 i b≤0).
6. Aquesta línia té quants punts de decisió?
if (a > 0 && b < 10 || c == 5)
El if + l'&& + el || = 3 decisions. Cadascuna pot fer curtcircuit i alterar el flux.
7. Què és un "camí independent"?
Cada nou camí ha d'incloure alguna cosa nova. Si dos camins recorren exactament les mateixes aristes, no són independents.
8. Si V(G) = 5, quantes proves cal com a mínim per cobrir tots els camins lògics?
V(G) és exactament el nombre mínim de camins independents que cal provar per cobrir tot el flux.
9. Una especificació diu: "El camp edat ha d'estar entre 0 i 120 inclosos". Quines classes d'equivalència s'identifiquen?
Cal identificar tant les classes vàlides com les invàlides per fer una cobertura completa.
10. Les proves de caixa blanca són pròpies de:
Cal accés al codi font. Qui prova ha de conèixer la implementació interna.
11. Quina mètrica indica que un mètode és massa complex i s'hauria de refactoritzar?
Una regla heurística molt acceptada: si V(G) > 10, el mètode és massa complex; cal extreure submètodes.
12. La combinació estàndard per a proves de caixa negra és:
Equivalència identifica els grans grups, límits cobreix els extrems on apareixen els bugs. Són dues tècniques complementàries de caixa negra.
